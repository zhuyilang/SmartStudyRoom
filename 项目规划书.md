# 智能校园自习室预约管理平台 — 项目规划书

---

## 一、技术栈选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端框架 | Spring Boot 2.7.x / 3.x | 主流Java后端框架，生态成熟 |
| ORM | MyBatis-Plus | 简化CRUD，分页、条件查询开箱即用 |
| 认证 | Spring Security + JWT | 无状态认证，适合前后端分离 |
| 数据库 | MySQL 8.0 | 关系型数据持久化 |
| 缓存 | Redis | 座位实时状态、会话管理、预约锁 |
| 定时任务 | Spring Task / XXL-Job | 超时释放、黑名单扫描 |
| 接口文档 | Knife4j (Swagger) | 前后端联调 |
| 构建工具 | Maven | 依赖管理 |
| 前端框架 | Vue 3 (Composition API) | 组件化开发 |
| UI组件库 | Element Plus | 表格、表单、弹窗等开箱即用 |
| 图表 | ECharts 5 | 热力图、折线图、柱状图 |
| 前端构建 | Vite | 快速冷启动 |
| HTTP客户端 | Axios | 请求拦截、统一错误处理 |
| 路由 | Vue Router 4 | SPA路由管理 |
| 状态管理 | Pinia | Vue 3 官方推荐 |

## 二、6人角色分工

### 👤 A — 组长（全栈/架构）
- 项目架构设计（目录结构、分层规范、公共组件）
- Git 仓库管理、分支策略、Code Review
- 后端公共模块：统一返回体、全局异常处理、JWT拦截器
- 用户登录/注册模块（后端+前端登录页）
- 最终联调集成 + 部署

### 👤 B — 后端·自习室管理
- 数据库 ER 图设计、建表 SQL
- 楼栋管理、楼层管理、自习室房间、座位排布 CRUD 接口
- 管理员权限控制
- 数据报表接口（使用率、热门时段）

### 👤 C — 后端·预约与座位管控
- 预约模块：选座、设置起止时间、单日次数限制
- 座位管控：签到/超时释放（定时任务）、取消预约
- 黑名单逻辑：爽约计数、禁约
- 实时看板数据接口（座位状态热力图数据）

### 👤 D — 前端·自习室管理 + 实时看板
- 管理后台页面：楼栋/楼层/房间/座位的增删改查
- 实时看板页面：热力图展示 + 自动刷新
- 前端路由与权限控制（管理员/学生）

### 👤 E — 前端·学生预约 + 数据报表
- 学生端首页：按校区/楼层筛选自习室
- 选座页面：座位网格可视化 + 预约时间选择
- 我的预约页面：签到、取消、历史记录
- 数据报表页面：图表展示

### 👤 F — 测试 + 文档 + 部署辅助
- 接口测试（Postman / JUnit）
- 前端功能测试用例编写
- 项目文档：接口文档核对、部署文档、用户手册
- 辅助组长进行部署（打包、环境配置）
- 数据库初始化数据准备

## 四、数据库核心表设计（建议）

```
campus          校区表          id, name
building        楼栋表          id, campus_id, name
floor           楼层表          id, building_id, floor_number
room            自习室表        id, floor_id, name, total_seats, open_time, close_time
seat            座位表          id, room_id, seat_number, status(0空闲/1占用/2暂离)
user            用户表          id, username, password, role(STUDENT/ADMIN), name
reservation     预约表          id, user_id, seat_id, start_time, end_time,
                                status(已预约/已签到/已取消/已超时/已完成),
                                sign_time, create_time
blacklist       黑名单表        id, user_id, violation_count, banned_until
```

## 七、关键风险与应对

| 风险 | 应对措施 |
|------|----------|
| 前端进度慢 | Element Plus组件丰富，优先用现成组件，后期再美化 |
| 座位并发冲突 | Redis分布式锁 / 数据库乐观锁，C同学重点攻关 |
| 定时任务不可靠 | 超时释放用Redis key过期监听+DB轮询双保险 |
| 联调时间不足 | 提前用Knife4j生成接口文档，前后端按文档并行开发 |
| 有人掉队 | 组长A和F随时补位，每天站会5分钟同步进度 |

---

> 预计总工时约 **15天 × 6人 = 90人天**，任务密度合理，第三周周五可交付完整的演示版本。
